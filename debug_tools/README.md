# debug_tools

**Developer-only utilities for analysing P2Poolv2 logs and data.**

These scripts were generated by Claude LLM and have not been reviewed
by the project authors. Use at your own discretion. They are not part
of the P2Poolv2 application and carry no correctness guarantees.

## Tools

### parse_share_lifecycle.py

Parses P2Poolv2 log files to show the lifecycle of each share:
received headers, block fetch, validation, candidate chain, confirmed
chain. Distinguishes locally-mined shares from peer-received shares.

```
python3 debug_tools/parse_share_lifecycle.py logs/p2pool.log
python3 debug_tools/parse_share_lifecycle.py logs/node1.log logs/node2.log
```

Pass two files to compare nodes side by side in two columns. Pipe
through `less -R` for colour support.

### share_latency.py

Computes share propagation and confirmation latencies across multiple
nodes by parsing their log files.

- Propagation latency: time from a share being mined at one node
  until it is validated at another node.
- Confirmation latency: time from a share first appearing at a node
  (mined or received) until it is promoted to the confirmed chain.

By default only confirmed chain shares are included. Use
`--include-uncles` to also include uncle shares.

```
python3 debug_tools/share_latency.py 'logs/node-*.log'
python3 debug_tools/share_latency.py --include-uncles logs/a.log logs/b.log
```

Prints per-pair and aggregate statistics (mean, stddev, min, max) with
the top 3 outliers for each grouping.

### verify_chain (Rust binary)

Traverses the entire confirmed share chain in a store.db and verifies
structural integrity. Opens the database in read-only mode so it is
safe to run against a live node's store.

Checks performed:
- Header and full block data exist for every confirmed share
- Transactions are non-empty (at least a coinbase)
- Metadata status is Confirmed with correct expected_height
- Chain work is monotonically increasing
- Parent linkage: each share's prev_share_blockhash matches the
  previous confirmed share
- Uncle count does not exceed MAX_UNCLES (3)
- Uncle headers and block data exist in the store
- No uncle is referenced by more than one confirmed nephew
- Uncles are not on the confirmed chain
- Uncle parent is an ancestor of the nephew (ancestor-type, not sibling)

```
just run /path/to/store.db
```

Returns exit code 0 on success, 1 if any errors are found.
